# RAG (Retrieval-Augmented Generation)

## 한계에서 출발한 아이디어

LLM은 학습 데이터 이후의 정보를 모른다. 사내 문서, 최신 뉴스, 개인 데이터처럼 **모델이 학습한 적 없는 지식**은 답할 수 없다.

파인튜닝으로 새 지식을 주입할 수 있지만 비용이 크고, 지식이 바뀔 때마다 재학습이 필요하다. RAG는 이 문제를 **"추론 시점에 문서를 검색해서 컨텍스트에 넣는다"** 는 방식으로 우회한다.

## 동작 원리

```
질문
  ↓
[Retriever] 벡터 DB에서 관련 문서 검색
  ↓
[Augment]   검색된 문서 + 질문을 합쳐 프롬프트 구성
  ↓
[Generator] LLM이 해당 컨텍스트 기반으로 답변 생성
```

핵심은 두 단계 분리다.

- **Retriever**: 문서를 벡터로 임베딩한 뒤 질문과의 유사도로 검색
- **Generator**: 검색 결과를 컨텍스트로 받아 답변 생성

## 왜 파인튜닝보다 나은 경우가 많은가

| | Fine-tuning | RAG |
|---|---|---|
| 지식 업데이트 | 재학습 필요 | DB만 교체 |
| 비용 | GPU 비용 높음 | 추론 비용만 |
| 출처 추적 | 불가 | 검색 문서 명시 가능 |
| 할루시네이션 | 여전히 발생 | 근거 문서로 억제 |

단, RAG도 검색 품질이 낮으면 엉뚱한 문서를 넣어 오히려 답이 나빠진다. **Retrieval 품질이 전체 성능의 병목**이다.

## 구성 요소

### Embedding Model
문서와 질문을 같은 벡터 공간에 매핑한다. OpenAI `text-embedding-3-small`, `bge-m3` 등이 자주 쓰인다.

### Vector Store
임베딩된 문서를 저장하고 유사도 검색을 수행한다. FAISS(로컬), Chroma, Pinecone, Weaviate 등.

### Chunking 전략
문서를 어떻게 쪼개느냐가 검색 품질을 좌우한다. 너무 짧으면 맥락 손실, 너무 길면 노이즈 증가. Recursive Character 분할, Semantic 분할 등 다양한 전략이 존재한다.

## Advanced RAG

기본 RAG의 한계를 극복하기 위한 확장 패턴들이 있다.

**HyDE (Hypothetical Document Embeddings)**
질문으로 가상의 답변을 먼저 생성하고, 그 답변으로 검색. 질문 자체보다 더 좋은 검색 결과를 얻는 경우가 많다.

**Re-ranking**
검색된 상위 N개 문서를 Cross-Encoder 모델로 재정렬. 벡터 유사도만으로 잡지 못하는 미묘한 관련성을 보완한다.

**Multi-hop Retrieval**
복잡한 질문을 서브 질문으로 분해하고 여러 번 검색. LangGraph의 순환 그래프와 결합해 구현하기도 한다.

## RAG vs Long Context

GPT-4의 128K, Gemini의 1M 컨텍스트 등 컨텍스트 윈도우가 계속 늘어나면서 "그냥 다 넣으면 되지 않나"는 질문이 생긴다.

현실적으로는 비용과 지연이 문제다. 수백만 토큰을 매번 전송하면 추론 비용이 선형으로 증가한다. RAG는 **필요한 것만 선택적으로 주입**한다는 점에서 여전히 실용적이다.
